# Contribuire alla documentazione di Mediterranean Atomic Tales - The Sunken Land
Grazie per il tuo interesse nel contribuire a __Mediterranean Atomic Tales - The Sunken Land (MAT:TSL)__.  
Questo progetto è prima di tutto un lavoro di __worldbuilding, documentazione e design__, ambientato in un universo ispirato a *Fallout*.

Contributi ponderati, coerenti e ben documentati sono sempre benvenuti.

---

# Filosofia del Progetto
Le linee quida principali sono:
- La __coerenza interna__ ha priorità sulla quantità;
- Il worldbuilding deve essere:
  - plausibile,
  - stratificato,
  - coerente con la timeline di Fallout;
- Ogni contributo dovrebbe __arricchire il mondo__, non solo aggiungere testo.

---

# Tipologie di Contributo Accettate
Puoi contribuire in diversi modi:

## Documentazione
- Espansione del worldbuilding
- Miglioramento di testi esistenti
- Correzione di refusi o errori concettuali
- Miglioramento della navigazione o struttura

## Lore e Ambientazione
- Nuove fazioni
- Luoghi, insediamenti, regioni
- Eventi storici coerenti con la timeline
- Cultura, società, economia postbellica

## Design di Gioco
- Meccaniche
- Sistemi di gioco
- Bilanciamento concettuale
- Idee per quest e gameplay

---

## Cosa NON accettiamo
- Fanfiction scollegata dal contesto
- Contenuti che contraddicono apertamente la lore di Fallout
- Aggiunte non documentate o senza contesto
- Materiale offensivo, discriminatorio o gratuito

---

# Struttura della Documentazione
- Tutti i file sono in __Markdown__
- Tutti i percorsi sono __minuscoli__
- Ogni directory deve avere un __index.md__ che contiene i link agli altri documenti della directory
- Se si aggiunge un documento ad una directory si deve aggiornare anche il relativo __index.md__
- I link devono essere __assoluti__ (es. "/MAT-TSL-documentation/...")

---

# Linee Guida di Scrittura
- Usa uno stile __chiaro e descrittivo__
- Evita muri di testo
- Preferisci sezioni, liste e titoli
- Specifica sempre:
  - _quando_ (timeline)
  - _dove_
  - _perché_

---

# Workflow di Contribuzione
1. Fork del repository
2. Crea un branch dedicato: git checkout -b feature/nome-breve
3. Fai commit chiari e atomici
4. Apri una Pull Request spiegando:
   - cosa aggiungi
   - perché è coerente
   - eventuali riferimenti interni

# Versioning e Release
Il progetto utilizza una forma adattata di [Semantic Versioning](https://semver.org/lang/it/).

- PATCH → correzioni e refactor
- MINOR → contenuti nuovi o sezioni complete
- MAJOR → stabilità dell’ambientazione

Consulta il file VERSIONING.md per i dettagli e il file CHANGELOG.md per visionare lo storico delle modifiche.
